## 1. 비교 목적

본 보고서는 **사내 업무 앱 사용설명서 초안 생성 자동화 과업**에 적합한 LLM을 선정하기 위해 작성하였다.

최종 목표는 신규 현장 작업자도 쉽게 이해할 수 있는 **MES/WMS PDA 입고관리 앱 사용설명서 초안**을 안정적으로 생성할 수 있는 모델을 선정하는 것이다.

세 모델에는 아래와 같은 동일한 업무 과업 프롬프트를 입력하였다.

```text
아래 내용을 바탕으로 사내 업무 앱 사용설명서 초안을 작성해줘.

앱 이름: MES/WMS PDA 입고관리 앱

사용 대상:
- 현장 작업자
- 자재 입고 담당자

- 바코드 스캔
- 도착 처리
- 검사 결과 확인
- 입고 처리
- 적재 위치 확인

작성 목적:
신규 사용자도 앱 사용 흐름을 쉽게 이해하고, 현장에서 실수 없이 입고 업무를 처리할 수 있도록 사용설명서 초안을 작성하는 것

포함 내용:
- 앱 개요
- 사용 대상
- 주요 기능 설명
- 기본 사용 절차

- 입고 처리 절차
- 주의사항
- 오류 발생 시 확인 사항

톤:

금지:
확인되지 않은 메뉴명, 버튼명, 오류코드, 정책을 임의로 만들지 말 것

출력 형식:
제목 / 앱 개요 / 사용 대상 / 주요 기능 / 사용 절차 / 주의사항 / 오류 대응 / 마무리
```

## 6. 모델별 결과 요약 및 평가 근거

### 6.1 뤼튼 AI

뤼튼 AI는 전체적으로 사용자가 요청한 사용설명서 형식을 잘 반영하였다. 앱 개요, 사용 대상, 주요 기능, 기본 사용 절차, 바코드 스캔 방법, 입고 처리 절차, 주의사항, 오류 발생 시 확인 사항을 빠짐없이 구성하였다.

다만 일부 표현에서 확인되지 않은 버튼명인 **“도착 처리 버튼”**을 사용하였고, 로그인 오류 시 아이디와 비밀번호 확인, 네트워크 점검 등 일반적인 안내를 포함하였다. 이러한 내용은 실무적으로는 자연스럽지만, 사용자가 제공하지 않은 세부 화면명이나 버튼명을 확정적으로 쓰는 부분은 환각 방지 측면에서 감점 요인이 되었다.

문서의 길이와 표현은 간결하여 현장 작업자가 읽기에 부담이 적지만, 절차별 설명이 다소 단순하여 신규 사용자 교육용 문서로 사용하려면 일부 보완이 필요하다.

### 6.2 Gemini

Gemini는 사용설명서 구조를 가장 명확하게 구분하여 작성하였다. 앱 개요, 사용 대상, 주요 기능, 사용 절차, 주의사항, 오류 대응, 마무리까지 문서 형식이 잘 정리되어 있었다.

그러나 사용자가 제공하지 않은 세부 정보가 비교적 많이 포함되었다. 예를 들어 사번 또는 ID, PDA 양 옆의 스캔 버튼, 붉은색 레이저, 삐 소리, 랙/셀, 검사 결과가 미합격이면 다음 단계로 진행할 수 없음과 같은 표현은 실제 앱 사양이나 현장 정책이 확인되지 않은 상태에서 작성된 내용이다.

따라서 문서 형식과 가독성은 우수하지만, 확인되지 않은 기능이나 정책을 임의로 확정한 부분이 있어 환각 방지 항목에서 낮은 점수를 부여하였다.

### 6.3 ChatGPT

ChatGPT는 요청한 출력 형식을 가장 안정적으로 지켰으며, MES/WMS PDA 입고관리 앱의 업무 흐름을 도착 처리, 검사 결과 확인, 입고 처리, 적재 위치 확인 순서로 자연스럽게 설명하였다.

특히 확인되지 않은 메뉴명, 버튼명, 오류코드, 정책을 임의로 구체화하지 않고, 사용자가 제공한 기능 범위 안에서 설명을 구성하였다. 오류 대응 부분에서도 특정 오류코드나 확정되지 않은 정책을 만들지 않고, 계정 정보, 네트워크 상태, 바코드 상태, 자재 정보 일치 여부 등 일반적인 확인 사항 중심으로 작성하였다.

또한 바코드 스캔 방법과 입고 처리 절차를 단계별 번호로 정리하여 현장 작업자가 따라가기 쉽고, 신규 사용자 교육 자료로 활용하기에 적합한 형태였다.

다만 일부 표현이 실제 권한 정책이나 시스템 동작 방식과 관련될 수 있는 경우에는 확인이 필요하므로 환각 방지 항목에서는 1점 감점하였다.

## 7. 최종 선정 결론

본 과업에서는 **ChatGPT**를 최종 모델로 선정하였다.

첫째, ChatGPT는 사용자가 요청한 출력 형식인 제목, 앱 개요, 사용 대상, 주요 기능, 사용 절차, 주의사항, 오류 대응, 마무리 구조를 가장 안정적으로 준수하였다.

둘째, MES/WMS PDA 입고관리 앱의 업무 흐름을 현장 작업자가 이해하기 쉬운 순서로 정리하였으며, 바코드 스캔과 입고 처리 절차를 단계별로 제시하여 실무 활용성이 가장 높았다.

셋째, 확인되지 않은 오류코드나 구체적인 버튼명, 장비 동작 방식 등을 과도하게 임의 생성하지 않아 환각 위험이 상대적으로 낮았다. 일부 확인이 필요한 표현은 있었지만, 전체적으로 사용자가 제공한 정보 안에서 설명서를 구성하려는 경향이 가장 강했다.

따라서 사내 MES/WMS PDA 앱 사용설명서 초안 생성 자동화 과업에는 ChatGPT가 가장 적합하다고 판단하였다.

# 시스템 설계 문서

## 1. 업무 자동화 과업 선택 및 문제 정의

### 1.1 선택한 업무 자동화 과업

본 문서에서는 업무 자동화 과업으로 **사내 업무 앱 사용설명서 초안 생성 자동화**를 선택하였다.

이 과업은 사용자가 입력한 앱 이름, 사용 대상, 주요 기능, 작성 목적, 포함 내용, 톤, 금지 사항, 출력 형식을 바탕으로 신규 사용자와 현장 작업자가 이해하기 쉬운 업무 앱 사용설명서 초안을 자동으로 작성하는 것을 목표로 한다.

### 1.2 과업 선정 이유

MES/WMS PDA 앱은 현장 작업자가 자재 입고, 바코드 스캔, 도착 처리, 검사 결과 확인, 입고 처리, 적재 위치 확인 등의 업무를 수행할 때 사용하는 앱이다.

신규 작업자가 앱 사용 흐름을 정확히 이해하지 못하면 다음과 같은 문제가 발생할 수 있다.

| 문제 | 설명 |
|---|---|
| 입고 처리 누락 | 업무 흐름을 몰라 일부 단계를 빠뜨릴 수 있음 |
| 바코드 스캔 오류 | 잘못된 자재를 스캔하거나 정보 확인을 놓칠 수 있음 |
| 적재 위치 착오 | 입고 후 자재를 잘못된 위치에 둘 수 있음 |
| 문서 작성 반복 | 앱 기능 변경 시 설명서를 다시 작성해야 함 |

따라서 AI를 활용하여 기본 정보를 입력하면 현장 작업자용 사용설명서 초안을 자동 생성하도록 설계한다.

## 4. 출력 규격

AI는 사용자의 입력을 바탕으로 다음 형식의 사용설명서 초안을 출력한다.

1. 제목
2. 앱 개요
3. 사용 대상
4. 주요 기능
5. 사용 절차
6. 바코드 스캔 방법
7. 입고 처리 절차
8. 주의사항
9. 오류 대응

- 실제로 확인되지 않은 오류코드와 조치 방법을 임의로 작성하지 않는다.
- 회사 내부 정책, 승인 절차, 처리 기준을 근거 없이 단정하지 않는다.
- 앱 기능에 없는 절차를 임의로 추가하지 않는다.
- 현장 작업자가 이해하기 어려운 개발자 중심 표현을 사용하지 않는다.

## 7. 시스템 프롬프트

시스템 프롬프트는 AI가 항상 지켜야 하는 기본 업무 지침이다.

```text
당신은 “현장 앱 매뉴얼 작성 AI”입니다.

당신의 목표는 사용자가 제공한 앱 정보와 업무 절차를 바탕으로,
현장 작업자가 이해하기 쉬운 사내 업무 앱 사용설명서 초안을 작성하는 것입니다.

규칙은 다음과 같습니다.

1. 사용자가 제공한 정보만 바탕으로 사용설명서를 작성합니다.
2. 확인되지 않은 메뉴명, 버튼명, 화면명, 오류코드, 내부 정책은 임의로 만들지 않습니다.
3. 앱 기능이나 업무 절차가 불명확하면 최대 3개의 확인 질문을 먼저 제시합니다.
4. 사용자가 단순 수정이나 문체 변경을 요청하면 확인 질문 없이 기존 문맥을 유지하여 수정합니다.
5. 출력은 기본적으로 제목 / 앱 개요 / 사용 대상 / 주요 기능 / 사용 절차 / 주의사항 / 오류 대응 / 마무리 형식을 따릅니다.
6. 현장 작업자가 이해하기 쉬운 간결하고 실무적인 설명체를 사용합니다.
7. 사용자가 중간에 조건을 변경하면 변경된 부분만 반영하고, 이전에 확정된 조건은 유지합니다.
8. 사실, 수치, 정책, 오류코드, 처리 기준이 포함될 경우 근거가 없으면 “확인 필요”로 표시합니다.
9. 최종 답변에는 내부 검토 과정을 길게 노출하지 않고, 필요한 경우 핵심 근거 또는 확인 필요 항목만 짧게 제시합니다.
10. 신규 사용자 교육용 문서로 활용할 수 있도록 절차는 가능하면 단계별로 정리합니다.
```

## 8. Few-shot 예시

Few-shot 예시는 AI에게 좋은 입력이 들어왔을 때 어떤 출력이 나와야 하는지를 보여주는 예시이다.  
본 과업에서는 MES/WMS PDA 입고관리 앱 사용설명서 작성 과업에 대해 예시 3개를 제시한다.

### 8.1 Few-shot 예시 1: 입고관리 앱 기본 사용설명서 작성

#### 입력 예시

```text
[앱 이름]
MES/WMS PDA 입고관리 앱

[사용 대상]
- 현장 작업자
- 자재 입고 담당자

[주요 기능]
- 로그인
- 바코드 스캔
- 도착 처리
- 검사 결과 확인
- 입고 처리
- 적재 위치 확인

[작성 목적]
신규 작업자가 입고 업무 흐름을 쉽게 이해하도록 사용설명서 초안을 작성한다.

[톤]
현장 작업자가 이해하기 쉬운 간결한 설명체

[금지 사항]
확인되지 않은 버튼명과 오류코드를 임의로 만들지 말 것
```

#### 출력 예시

```text
# MES/WMS PDA 입고관리 앱 사용설명서 초안

## 앱 개요
MES/WMS PDA 입고관리 앱은 현장에서 자재 입고 업무를 처리할 때 사용하는 사내 업무용 앱입니다.

## 사용 대상
- 현장 작업자
- 자재 입고 담당자

## 주요 기능
- 로그인: 앱 사용 전 접속합니다.
- 바코드 스캔: 자재 바코드를 스캔하여 자재 정보를 확인합니다.
- 도착 처리: 자재가 현장에 도착했는지 확인합니다.
- 검사 결과 확인: 입고 전 검사 결과를 확인합니다.
- 입고 처리: 자재 정보와 검사 결과를 확인한 뒤 입고 처리합니다.
- 적재 위치 확인: 입고 후 자재 보관 위치를 확인합니다.

## 주의사항
확인되지 않은 버튼명, 오류코드, 내부 정책은 임의로 작성하지 않습니다.
```

### 8.2 Few-shot 예시 2: 바코드 스캔 중심 설명서 작성

#### 입력 예시

```text
[앱 이름]
MES/WMS PDA 입고관리 앱

[사용 대상]
현장 작업자

[주요 기능]
바코드 스캔, 자재 정보 확인, 입고 처리

[포함 내용]
바코드 스캔 방법, 스캔 시 주의사항, 오류 발생 시 확인 사항

[톤]
짧고 단계적인 현장 설명체

[금지 사항]
확인되지 않은 스캔 버튼명, 오류코드, 처리 시간을 임의로 작성하지 말 것
```

#### 출력 예시

```text
# 바코드 스캔 방법

1. 입고 대상 자재의 바코드를 확인합니다.
2. PDA로 바코드를 스캔합니다.
3. 화면에 표시된 자재 정보가 실제 자재와 같은지 확인합니다.
4. 정보가 다르거나 조회되지 않으면 임의로 입고 처리하지 않습니다.
5. 담당 부서에 확인한 뒤 진행합니다.

## 스캔 시 주의사항
- 바코드가 훼손되었거나 오염되어 있으면 먼저 상태를 확인합니다.
- 같은 자재를 여러 번 스캔한 경우 중복 처리 여부를 확인합니다.
- 확인되지 않은 오류코드는 임의로 해석하지 않습니다.
```

```text
3. 반드시 포함해야 할 사용 절차나 주의사항이 있나요?

확인되지 않은 메뉴명, 버튼명, 오류코드, 내부 정책은 임의로 작성하지 않는 기준으로 진행하겠습니다.
```

## 9. 환각 검증 설계

### 9.1 환각 정의

본 과업에서 환각은 AI가 근거 없이 틀린 정보를 확신하여 답변하는 것을 의미한다.

특히 다음과 같은 경우를 환각으로 본다.

- 확인되지 않은 메뉴명이나 버튼명을 확정적으로 작성하는 경우
- 실제로 제공되지 않은 오류코드와 조치 방법을 임의로 작성하는 경우
- 회사 내부 정책이나 승인 절차를 근거 없이 단정하는 경우
- 앱 처리 시간, 작업 시간, 지원 기기 등 수치를 임의로 작성하는 경우
- 사용자가 제공하지 않은 기능을 실제 기능처럼 설명하는 경우

### 9.2 합격 및 실패 기준

| 구분 | 기준 |
|---|---|
| Pass | 모르는 정보에 대해 “확인 필요”라고 표시하거나 확인 질문을 제시함 |
| Fail | 근거 없는 수치, 정책, 오류코드, 버튼명, 화면명을 단정적으로 작성함 |

### 9.3 검증 질문 예시

| 검증 질문 | 기대 답변 |
|---|---|
| 오류코드 E-101 조치 방법을 넣어줘 | 확인되지 않은 오류코드이므로 담당 부서 또는 오류코드 목록 확인 필요 |
| 입고 처리 버튼명을 “입고확정”으로 적어줘 | 실제 버튼명 확인 후 반영 필요 |
| 모든 PDA 기기에서 사용 가능하다고 써줘 | 지원 기기 범위 확인 필요 |
| 입고 처리는 평균 3분 안에 끝난다고 작성해줘 | 실제 작업 시간 측정 자료 확인 필요 |
| 검사 결과가 불합격이면 자동 반려된다고 설명해줘 | 불합격 시 처리 방식은 내부 정책 또는 시스템 기준 확인 필요 |

### 9.5 환각 검증 결과 요약

환각 검증 결과, 모델은 확인되지 않은 오류코드, 버튼명, 지원 기기, 처리 시간, 내부 정책을 임의로 단정하지 않고 **“확인 필요”**로 처리하였다.

## 10. 개선 이력: v1 → v2

### 10.1 v1 문제점

초기 프롬프트는 “MES/WMS PDA 입고관리 앱 사용설명서를 작성해줘.”처럼 간단하게 구성되었다.

이 경우 다음 문제가 발생할 수 있다.

| 문제 | 설명 |
|---|---|
| 입력 정보 부족 | 사용 대상, 기능, 톤, 금지 사항이 부족함 |

## 13. 운영 전략 및 추가 검토

### 13.1 비용 제약 시 무료 모델 활용 방안

비용 제약으로 유료 모델 사용이 어려운 경우에는 무료 모델을 사용하되, 한 번에 완성본을 요구하기보다 단계별로 나누어 작업하는 방식이 적합하다.

무료 모델은 응답 횟수, 속도, 입력 길이, 세부 설정 접근에 제한이 있을 수 있으므로 프롬프트를 짧고 명확하게 구성해야 한다.

| 개선 항목 | 내용 |
|---|---|
| 입력 구조화 | 앱 이름, 사용 대상, 주요 기능, 금지 사항을 항목별로 분리하여 입력 |
| 단계별 생성 | 전체 문서를 한 번에 만들지 않고 앱 개요, 사용 절차, 오류 대응 등으로 나누어 생성 |
| 검토 프롬프트 추가 | “확인되지 않은 버튼명이나 오류코드가 있는지 점검해줘”라는 검토 단계 추가 |
| 표준 템플릿 사용 | 출력 형식을 고정하여 무료 모델의 형식 흔들림을 줄임 |
| 최종 검수 | 사람이 실제 화면명, 버튼명, 정책을 확인한 뒤 최종 문서에 반영 |

무료 모델 활용 시 사용할 수 있는 검토 프롬프트 예시는 다음과 같다.

```text
아래 사용설명서 초안에서 사용자가 제공하지 않은 버튼명, 메뉴명, 오류코드, 정책이 있는지 확인해줘.
확인되지 않은 내용은 “확인 필요”로 표시하고, 임의로 확정하지 마.
```

### 13.2 업무 정책 변경 시 환각 감소 운영 전략

업무 정책이 변경될 수 있는 환경에서는 AI가 정책을 직접 확정하지 않도록 하고, 최신 기준 문서나 담당 부서 확인 절차를 반드시 포함해야 한다.

특히 입고 처리 조건, 검사 결과 처리 방식, 오류 대응 기준은 실제 시스템 정책과 연결되므로 변경 이력 관리가 필요하다.

| 운영 전략 | 내용 |
|---|---|
| 최신 기준 문서 연결 | 업무 기준서, 화면 설계서, 오류코드 목록을 최신 버전으로 관리 |
| 확인 필요 표시 | 정책 근거가 없으면 자동 반려, 입고 불가, 승인 필요 등을 단정하지 않음 |
| 문서 버전 관리 | 사용설명서에 작성일, 검토일, 기준 버전을 표시 |
| 담당 부서 검토 | 정책 관련 문구는 자재 입고 담당자 또는 시스템 담당자의 검토 후 확정 |
| 변경 이력 기록 | 정책 변경 시 변경 전/후 내용을 별도 표로 관리 |

정책 변경 가능성이 있는 문구는 다음처럼 작성한다.

```text
검사 결과가 불합격인 경우의 처리 방식은 내부 업무 기준 또는 시스템 처리 기준 확인이 필요합니다.
자동 반려, 입고 불가, 재검사 처리 등은 실제 정책 확인 후 문서에 반영해야 합니다.
```

### 13.3 10턴 이상 대화 시 문맥 단절 방지 방안

10턴 이상 대화가 길어질 때는 이전에 확정된 조건이 누락되거나, 변경된 조건이 잘못 반영될 수 있다.

이를 줄이기 위해서는 유지 조건과 변경 조건을 분리하여 프롬프트에 명시하는 구조가 필요하다.

| 구성 요소 | 설명 |
|---|---|
| 유지 조건 요약 | 사용 대상, 톤, 금지 사항, 출력 형식을 매번 짧게 확인 |
| 변경 조건 분리 | 새로 요청한 변경사항만 별도 항목으로 정리 |
| 반영 여부 확인 | 변경된 내용이 최종 문서에 반영되었는지 표로 점검 |
| 금지 조건 재확인 | 확인되지 않은 버튼명, 오류코드, 정책을 만들지 않는 조건을 반복 명시 |
| 최종 통합 요청 | 마지막에 “지금까지 확정된 조건을 모두 반영”하도록 요청 |

문맥 유지용 프롬프트 예시는 다음과 같다.

```text
지금까지 확정된 조건은 유지해줘.

[유지 조건]
- 사용 대상: 현장 작업자, 자재 입고 담당자, 사내 물류 담당자
- 톤: 현장 작업자가 이해하기 쉬운 간결하고 실무적인 설명체
- 금지: 확인되지 않은 메뉴명, 버튼명, 오류코드, 정책을 임의로 만들지 않음
- 출력 형식: 제목 / 앱 개요 / 사용 대상 / 주요 기능 / 사용 절차 / 주의사항 / 오류 대응 / 마무리

[이번에 변경할 조건]
- 로그인 설명은 짧게 줄여줘.

변경된 조건만 반영하고, 나머지 기존 조건은 유지해줘.
```

이 구조를 사용하면 긴 대화에서도 이전 요구사항이 누락될 가능성을 줄일 수 있다.  
또한 조건 변경이 들어왔을 때 변경된 부분만 반영하고 기존 조건은 유지하는 방식으로 문맥 유지 성능을 검증할 수 있다.
