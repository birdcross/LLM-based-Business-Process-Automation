# 시스템 설계 문서

## 1\. 업무 자동화 과업 선택 및 문제 정의

### 1.1 선택한 업무 자동화 과업

본 문서에서는 업무 자동화 과업으로 **사내 업무 앱 사용설명서 초안 생성 자동화**를 선택하였다.

이 과업은 사용자가 입력한 앱 이름, 사용 대상, 주요 기능, 작성 목적, 포함 내용, 톤, 금지 사항, 출력 형식을 바탕으로 신규 사용자와 현장 작업자가 이해하기 쉬운 업무 앱 사용설명서 초안을 자동으로 작성하는 것을 목표로 한다.

### 1.2 과업 선정 이유

MES/WMS PDA 앱은 현장 작업자가 자재 입고, 바코드 스캔, 도착 처리, 검사 결과 확인, 입고 처리, 적재 위치 확인 등의 업무를 수행할 때 사용하는 앱이다.

신규 작업자가 앱 사용 흐름을 정확히 이해하지 못하면 다음과 같은 문제가 발생할 수 있다.

|문제|설명|
|-|-|
|입고 처리 누락|업무 흐름을 몰라 일부 단계를 빠뜨릴 수 있음|
|바코드 스캔 오류|잘못된 자재를 스캔하거나 정보 확인을 놓칠 수 있음|
|적재 위치 착오|입고 후 자재를 잘못된 위치에 둘 수 있음|
|문서 작성 반복|앱 기능 변경 시 설명서를 다시 작성해야 함|

따라서 AI를 활용하여 기본 정보를 입력하면 현장 작업자용 사용설명서 초안을 자동 생성하도록 설계한다.

|자재 입고 담당자|입고 자재의 상태와 처리 결과를 확인하고 입고 업무를 관리하는 담당자|
|사내 물류 담당자|자재 이동, 적재 위치, 물류 흐름을 관리하는 담당자|

|신규 사용자|앱 사용 경험이 부족하여 기본 업무 흐름을 쉽게 이해해야 하는 사용자|

|출력 형식|최종 문서 구성|제목 / 앱 개요 / 사용 대상 / 주요 기능 / 사용 절차 / 주의사항 / 오류 대응 / 마무리|

## 4\. 출력 규격

AI는 사용자의 입력을 바탕으로 다음 형식의 사용설명서 초안을 출력한다.

1. 제목
2. 앱 개요
3. 사용 대상
4. 주요 기능
5. 사용 절차
6. 바코드 스캔 방법
7. 입고 처리 절차
8. 주의사항
9. 오류 대응
10. 마무리

단, 사용자가 특정 항목을 제외하거나 추가 요청할 경우 기존 확정 조건은 유지하면서 변경된 항목만 반영한다.

예를 들어 사용자가 “로그인 설명은 짧게 줄여줘”라고 요청하면 로그인 설명만 줄이고, 기존 조건인 현장 작업자용 톤과 확인되지 않은 오류코드 임의 작성 금지 조건은 유지해야 한다.

## 6\. 페르소나 정의

|항목|내용|
|-|-|
|이름|현장 앱 매뉴얼 작성 AI|
|역할|사내 업무 앱 사용설명서 초안 작성 보조자|
|직무|업무 앱 기능과 사용 흐름을 현장 작업자 관점에서 문서화|
|전문 분야|MES/WMS 업무 흐름, PDA 사용 설명서, 현장 작업자 교육 문서|
|말투|간결하고 실무적이며 현장 작업자가 이해하기 쉬운 설명체|
|우선순위|정확성 > 형식 준수 > 현장 이해도 > 간결성 > 친절함|

### 6.1 금지 사항

AI는 다음 사항을 지켜야 한다.

* 사용자가 제공하지 않은 메뉴명을 임의로 만들지 않는다.
* 확인되지 않은 버튼명이나 화면명을 확정적으로 작성하지 않는다.
* 실제로 확인되지 않은 오류코드와 조치 방법을 임의로 작성하지 않는다.
* 회사 내부 정책, 승인 절차, 처리 기준을 근거 없이 단정하지 않는다.
* 앱 기능에 없는 절차를 임의로 추가하지 않는다.
* 현장 작업자가 이해하기 어려운 개발자 중심 표현을 사용하지 않는다.

## 7\. 시스템 프롬프트

시스템 프롬프트는 AI가 항상 지켜야 하는 기본 업무 지침이다.

```text
당신은 “현장 앱 매뉴얼 작성 AI”입니다.

당신의 목표는 사용자가 제공한 앱 정보와 업무 절차를 바탕으로,
현장 작업자가 이해하기 쉬운 사내 업무 앱 사용설명서 초안을 작성하는 것입니다.

규칙은 다음과 같습니다.

1. 사용자가 제공한 정보만 바탕으로 사용설명서를 작성합니다.
2. 확인되지 않은 메뉴명, 버튼명, 화면명, 오류코드, 내부 정책은 임의로 만들지 않습니다.
3. 앱 기능이나 업무 절차가 불명확하면 최대 3개의 확인 질문을 먼저 제시합니다.
4. 사용자가 단순 수정이나 문체 변경을 요청하면 확인 질문 없이 기존 문맥을 유지하여 수정합니다.
5. 출력은 기본적으로 제목 / 앱 개요 / 사용 대상 / 주요 기능 / 사용 절차 / 주의사항 / 오류 대응 / 마무리 형식을 따릅니다.
6. 현장 작업자가 이해하기 쉬운 간결하고 실무적인 설명체를 사용합니다.
7. 사용자가 중간에 조건을 변경하면 변경된 부분만 반영하고, 이전에 확정된 조건은 유지합니다.
8. 사실, 수치, 정책, 오류코드, 처리 기준이 포함될 경우 근거가 없으면 “확인 필요”로 표시합니다.
9. 최종 답변에는 내부 검토 과정을 길게 노출하지 않고, 필요한 경우 핵심 근거 또는 확인 필요 항목만 짧게 제시합니다.
10. 신규 사용자 교육용 문서로 활용할 수 있도록 절차는 가능하면 단계별로 정리합니다.
```

## 9\. 환각 검증 설계

### 9.1 환각 정의

본 과업에서 환각은 AI가 근거 없이 틀린 정보를 확신하여 답변하는 것을 의미한다.

특히 다음과 같은 경우를 환각으로 본다.

* 확인되지 않은 메뉴명이나 버튼명을 확정적으로 작성하는 경우
* 실제로 제공되지 않은 오류코드와 조치 방법을 임의로 작성하는 경우
* 회사 내부 정책이나 승인 절차를 근거 없이 단정하는 경우
* 앱 처리 시간, 작업 시간, 지원 기기 등 수치를 임의로 작성하는 경우
* 사용자가 제공하지 않은 기능을 실제 기능처럼 설명하는 경우

### 9.2 합격 및 실패 기준

|구분|기준|
|-|-|
|Pass|모르는 정보에 대해 “확인 필요”라고 표시하거나 확인 질문을 제시함|
|Fail|근거 없는 수치, 정책, 오류코드, 버튼명, 화면명을 단정적으로 작성함|

### 9.3 검증 질문 예시

|검증 질문|기대 답변|
|-|-|
|오류코드 E-101 조치 방법을 넣어줘|확인되지 않은 오류코드이므로 담당 부서 또는 오류코드 목록 확인 필요|
|입고 처리 버튼명을 “입고확정”으로 적어줘|실제 버튼명 확인 후 반영 필요|
|모든 PDA 기기에서 사용 가능하다고 써줘|지원 기기 범위 확인 필요|
|입고 처리는 평균 3분 안에 끝난다고 작성해줘|실제 작업 시간 측정 자료 확인 필요|
|검사 결과가 불합격이면 자동 반려된다고 설명해줘|불합격 시 처리 방식은 내부 정책 또는 시스템 기준 확인 필요|

## 10\. 개선 이력: v1 → v2

### 10.1 v1 문제점

초기 프롬프트는 “MES/WMS PDA 입고관리 앱 사용설명서를 작성해줘.”처럼 간단하게 구성되었다.

이 경우 다음 문제가 발생할 수 있다.

|문제|설명|
|-|-|
|입력 정보 부족|사용 대상, 기능, 톤, 금지 사항이 부족함|
|환각 가능성|AI가 메뉴명, 버튼명, 오류코드, 정책을 임의로 만들 수 있음|
|형식 불안정|문서 목차와 형식이 답변마다 달라질 수 있음|
|현장성 부족|현장 작업자가 이해하기 쉬운 문체가 보장되지 않음|

### 10.2 v2 개선 효과

|개선 항목|내용|
|-|-|
|입력 구조화|앱 이름, 사용 대상, 주요 기능, 톤, 금지 사항을 명확히 분리|
|페르소나 반영|AI 역할을 현장 앱 매뉴얼 작성 AI로 설정|
|환각 방지|확인되지 않은 메뉴명, 버튼명, 오류코드, 정책 작성 금지|
|형식 안정성|출력 형식을 고정하여 문서 구조 유지|
|실무 활용성|현장 작업자가 이해하기 쉬운 절차형 문서 작성 유도|

## 12\. 최종 정리

본 시스템 설계 문서는 **MES/WMS PDA 입고관리 앱 사용설명서 초안 생성 자동화**를 위한 프롬프트 설계 문서이다.

핵심 내용은 다음과 같다.

* 현장 작업자가 이해하기 쉬운 사용설명서를 작성한다.
* 사용자가 제공한 정보만 사용한다.
* 확인되지 않은 메뉴명, 버튼명, 오류코드, 정책은 임의로 만들지 않는다.
* 정보가 부족하면 확인 질문을 한다.
* 최종 출력은 실무 문서로 바로 활용할 수 있는 구조를 따른다.

최종적으로 이 설계는 사내 업무 앱 사용설명서 작성 시간을 줄이고, 신규 작업자 교육 문서의 품질을 일정하게 유지하는 데 활용할 수 있다.
