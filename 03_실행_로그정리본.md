|항목|내용|

|과업명|MES/WMS PDA 입고관리 앱 사용설명서 초안 생성|

|실행 목적|시스템 프롬프트와 입력 템플릿을 적용했을 때, AI가 현장 작업자용 앱 사용설명서를 안정적으로 작성하고 수정 요청을 문맥에 맞게 반영하는지 확인|

## 3\. 안전/윤리 준수 사항

|항목|준수 내용|
|-|-|
|폭력적/선정적 콘텐츠|폭력적 또는 선정적 콘텐츠 생성을 시도하지 않았다.|
|개인정보|실명, 전화번호, 주소, 계정정보 등 개인정보를 프롬프트와 로그에 포함하지 않았다.|
|사실/정책/수치 검증|오류코드, 처리 시간, 내부 정책, 버튼명 등 확인되지 않은 정보는 단정하지 않고 “확인 필요”로 표시하도록 설계하였다.|
|업무 보안|실제 회사 내부 정책이나 미확인 시스템 정보를 임의로 작성하지 않도록 제한하였다.|

## 4\. 적용한 시스템 프롬프트



당신은 “현장 앱 매뉴얼 작성 AI”입니다.

당신의 목표는 사용자가 제공한 앱 정보와 업무 절차를 바탕으로, 현장 작업자가 이해하기 쉬운 사내 업무 앱 사용설명서 초안을 작성하는 것입니다.

반드시 지킬 규칙은 다음과 같습니다.

1. 사용자가 제공한 정보만 바탕으로 작성합니다.
2. 확인되지 않은 메뉴명, 버튼명, 화면명, 오류코드, 회사 정책, 처리 기준은 임의로 만들지 않습니다.
3. 정보가 부족하면 최대 3개의 확인 질문을 먼저 제시합니다.
4. 단순한 문장 수정, 톤 변경, 형식 변경 요청이면 확인 질문 없이 기존 문맥을 유지하여 수정합니다.
5. 출력은 기본적으로 제목 / 앱 개요 / 사용 대상 / 주요 기능 / 사용 절차 / 주의사항 / 오류 대응 / 마무리 형식을 따릅니다.
6. 현장 작업자가 이해하기 쉬운 간결하고 실무적인 설명체로 작성합니다.
7. 신규 사용자 교육용 문서로 활용할 수 있도록 절차는 단계별로 정리합니다.
8. 사용자가 중간에 조건을 변경하면 변경된 부분만 반영하고 이전에 확정된 조건은 유지합니다.
9. 사실, 수치, 정책, 오류코드, 처리 기준이 포함될 경우 근거가 없으면 “확인 필요”로 표시합니다.
10. 내부 검토 과정은 최종 답변에 길게 노출하지 않고, 필요한 경우 핵심 근거 또는 확인 필요 항목만 짧게 제시합니다.

## 5\. 대화 로그 요약

대화 로그는 그대로 덤프하지 않고, 가독성을 위해 주요 목적과 수정 흐름을 기준으로 정리하였다.



|턴|사용자 요청 요약|AI 응답 요약|반영된 조건|
|-:|-|-|-|
|1|앱 정보와 출력 형식을 제공하고 사용설명서 초안 요청|제목, 앱 개요, 사용 대상, 주요 기능, 사용 절차, 주의사항, 오류 대응, 마무리 형식으로 초안 작성|기본 출력 형식 반영|
|2|현장 작업자가 이해하기 쉽게 수정 요청|문장을 더 짧고 쉬운 표현으로 수정|현장 작업자용 문체 강화|
|3|입고 처리 절차를 단계별 번호로 상세화 요청|입고 대상 확인부터 처리 결과 확인까지 단계별 절차 제시|절차형 문서 강화|
|4|바코드 스캔 방법과 주의사항 추가 요청|바코드 상태 확인, 스캔, 정보 확인, 오류 시 확인 절차 추가|바코드 스캔 항목 보강|
|5|확인되지 않은 오류코드 임의 생성 금지 요청|오류 대응을 표로 정리하고 실제 오류 목록 확인 필요로 표시|환각 방지|
|6|로그인 설명 축소 요청|로그인 설명을 짧게 수정|불필요한 설명 축소|
|7|PDA 화면 기준 설명 요청, 버튼명 임의 생성 금지|화면 흐름 기준으로 설명하되 실제 화면명/버튼명은 확인 필요로 처리|미확인 버튼명 방지|
|8|신규 작업자 교육용 문서 느낌 요청|기본 업무 흐름과 신규 작업자 확인 사항으로 재구성|교육 문서 형태 반영|
|9|표가 필요한 부분은 표로 정리 요청|주요 기능과 오류 대응을 표 형식으로 정리|가독성 개선|
|10|검사 결과 불합격 시 자동 반려 여부 확인|내부 정책 확인 필요로 답변하고 단정 표현 금지|정책 환각 방지|
|11|지금까지 조건을 반영한 최종본 요청|최종 사용설명서 형식으로 통합 정리|전체 조건 유지|

## 7\. 문제 발생 지점 및 수정 결과

|구분|내용|
|-|-|
|문제 발생 지점 1|사용자가 PDA 화면 기준 설명을 요청했으나 실제 화면명과 버튼명이 제공되지 않음|
|수정 결과 1|실제 화면명과 버튼명을 임의로 만들지 않고 “화면 확인 후 작성 필요”로 처리|
|문제 발생 지점 2|사용자가 검사 결과 불합격 시 자동 반려 여부를 질문함|
|수정 결과 2|내부 정책이 확인되지 않았으므로 “자동 반려”라고 단정하지 않고 확인 필요로 처리|
|문제 발생 지점 3|중간에 로그인 설명 축소, 표 형식 적용, 신규 작업자 교육용 문서 등 조건이 변경됨|
|수정 결과 3|변경된 부분만 반영하고 기존 조건인 현장 작업자용 톤, 환각 방지, 출력 형식은 유지|
